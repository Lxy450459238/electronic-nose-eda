## 四、multi.py 修改详情

**来源：** `EDA_MULTI_合3.py`

### 4.1 导入变更（关键升级）

```python
# 旧
from EDA_CONFIG_universal3 import UniversalConfig as Config

# 新
from config import UniversalConfig as Config
```

这是最重要的修改。原 `multi.py` 引用的是 config3（该版本不支持 `env_config`、`env_name_map`、`ENV_VALID_RANGES` 等新版特性），现已升级到与 single.py 一致的 config5。

### 4.2 兼容性确认

multi.py 使用的所有 Config 属性在 config5 中均存在：

| 使用的属性 | config3 | config5 |
|-----------|---------|---------|
| `detect_version()` | ✓ | ✓ |
| `extract_sensor_data()` | ✓ | ✓ |
| `get_sensor_name()` | ✓ (仅返回 f'sensor{i}') | ✓ (返回真实传感器名) |
| `STABILITY_CV_STABLE/UNSTABLE` | ✓ | ✓ |
| `DISCRIMINATION_SNR_WEIGHT/F_WEIGHT` | ✓ | ✓ |
| `DISCRIMINATION_SCORE_*/P_VALUE_*` | ✓ | ✓ |
| `OUTLIER_*` | ✓ | ✓ |
| `CORRELATION_*` | ✓ | ✓ |
| `REPORT_TOP_*` | ✓ | ✓ |
| `VIS_*` | ✓ | ✓ |

**改进点：** config5 的 `get_sensor_name()` 返回真实传感器型号名称（如 "00_TGS813"），而 config3 仅返回 "sensor0" 等占位名。这使得 multi.py 的报告和图表中传感器标签更加可读。

### 4.3 删除的代码

| `_calculate_discrimination()` | `is_new_version` 检测及 `machine_type` 打印（~3行） | 仅用于日志显示，config5 通过 `version_config['name']` 已提供更丰富的版本信息 |

### 4.4 注释清理

与 single.py 同理，清理了全局的 `# [NEW]`、`# [TOOL]` 等迭代标记，保留功能性注释。

### 4.5 功能完整性

- 所有多文件分析逻辑完全保留

## 七、v6 修复核实并入（2026.06.08 追加）

在初始整理（基于 v5）完成后，发现存在 v6 版本 `EDA_SINGLE_合6.py`，该版本针对王宇老师于 4.28 提出的三个问题进行了修复。经 diff 对比确认，v6 相对于 v5 仅有 single.py 发生变化（config 无变化）。

以下三项修复已核实并应用到 `single.py` 最终版：

### 7.1 日期解析错误修复

**问题：** 文件名如 `yangji1-034-huangqin(26)-2408-830-20211013_20260306094501.txt` 中，旧版 `_parse_metadata()` 简单按 `-` 分割取第一段作为日期，导致日期显示为 "yangji1"。

**修复：**
- 新增 `import re`
- 重写 `_parse_metadata()`，使用正则 `r'20\d{6}'` 智能匹配文件名中任意位置的8位连续日期数字
- 修复后该文件日期正确解析为 "20211013"

**涉及行号：** 第15行（import）、第131-154行（方法重写）

### 7.2 不稳定传感器描述文字修改

**问题：** 报告中极差≥120的传感器标注为"剧烈起伏/单调漂移"，老师建议统一为更规范的描述。

**修复：** 将全部3处 `"剧烈起伏/单调漂移"` 替换为 `"基线波动幅度过大"`

**涉及位置：**
- `analyze_baseline()` 控制台输出
- `generate_report()` 控制台输出
- `save_text_report()` 文本报告输出

### 7.3 传感器漂移警告具体化

**问题：** 清洗阶段漂移警告仅笼统提示"P18阶段可能存在传感器漂移"，未列出具体传感器。

**修复：** 警告信息改为 `f"[WARNING] {clean_stage}阶段存在传感器漂移: {drift_str}"`，其中 `drift_str` 包含具体传感器名称列表（超过5个时追加"等"）。

**涉及位置：** `analyze_cleaning()` 方法

### v6 变更来源确认

- 源文件：`EDA综合版本V2/EDA_SINGLE_合6.py`
- 对比基准：`EDA综合版本V2/EDA_SINGE_合5.py`
- 差异确认：通过 `diff -u` 完整对比，确认上述三项为 v6 的全部功能性修改（其余为注释清理和格式调整，已在初始整理中完成）

## 八、multi.py 同步升级（2026.06.08 追加）

在核实 v6 修复时发现 `multi.py`（源自 `EDA_MULTI_合3.py`）自 v3 后未跟随 `single.py` 的迭代。经系统性对比，确认以下改进需要同步应用到 `multi.py`。

### 8.1 日期解析 Bug 修复

**问题：** `_parse_filename()` 与 `single.py` v5 存在相同 bug —— 简单取 `parts[0]` 作为日期，文件名含前缀时日期错误（如 "yangji1"）。

**修复：**
- 新增 `import re`
- 重写 `_parse_filename()`，使用正则 `r'20\d{6}'` 智能提取日期

**涉及位置：** 第16行（import）、第128-153行（方法重写）

### 8.2 首行零值跳过逻辑

**问题：** `load_and_extract_data()` 直接读取数据，未检查前几行核心传感器是否含0（硬件通信未就绪），可能导致含通信故障行的数据混入稳态计算。

**修复：** 在提取传感器数据后，检查前 `DATA_SKIP_ROWS_MAX`（3）行核心传感器（前 `DATA_SKIP_CHECK_COLS` 列）是否含0值，含0则跳过该行并对 `sensor_data` 和 `stage_column` 执行同步裁切。

**涉及位置：** `load_and_extract_data()` 方法

### 8.3 环境通道体检模块

**问题：** `multi.py` 完全没有提取或检查环境/物理通道数据（供电电压、温度、湿度、气压），批处理时无法发现因环境异常导致的系统性坏样本。

**修复：**
- 在 `__init__` 中新增 `self.env_warnings = []`
- 在 `load_and_extract_data()` 处理第一个文件时调用 `Config.extract_env_data()` 提取环境数据
- 新增 `analyze_environment()` 方法，对电源、温度、湿度、气压进行绝对区间+STD波动判断
- 在 `run_full_analysis()` 流程中插入环境体检步骤（加载数据后、判断多样性前）
- 在 `generate_report()` 文本报告中新增"硬件体检"章节

**涉及位置：** `__init__`、`load_and_extract_data()`、新增 `analyze_environment()`、`run_full_analysis()`、`generate_report()`

### 8.4 测试验证

分别在旧版37传感器数据（多样性样本2，15个文件）和新版49传感器数据（多样性样本2_重邮，9个文件）上运行测试，确认：

- ✅ 日期解析正确（旧版文件名 `20240510-302-bingtong-30-1.txt` → 日期 `20240510`）
- ✅ 硬件体检章节出现在报告中
- ✅ 旧版识别为 "旧版37传感器"，新版识别为 "精简版49纯气体传感器"
- ✅ 所有可视化图表正常生成
- ✅ 传感器名称映射正确（真实名称，非纯数字索引）
