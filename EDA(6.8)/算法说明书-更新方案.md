## 二、具体新增内容草案

### 2.1 硬件版本自动识别（新增 §2.3）

```
系统通过解析数据文件的列数自动判定电子鼻硬件版本：
- 列数 ≥ 83 → 新版49传感器（精简版49纯气体传感器）
- 列数 ≥ 51 → 旧版37传感器
- 列数 < 51 → 报错退出

识别后自动加载对应的传感器名称映射表、列切片配置和环境通道配置。
```

### 2.2 四维基线稳定性判别准则（替换 §3.1.2）

当前代码实现的四维判定规则，任一维度触碰红线即判定为不稳定：

**维度A — 大均值CV规则：**
当传感器基线均值 |μ| ≥ 50 时，使用 CV ≥ 0.15 判定。适用于基准读数较高的传感器。

**维度B — 小均值STD兜底规则：**
当传感器基线均值 |μ| < 50 时，切换为绝对STD ≥ 20.0 判定。防止差分输出/归零传感器的CV数学爆炸导致误杀。

**维度C — 全局大基数STD规则：**
任何传感器 STD ≥ 30.0 直接判为不稳定。针对 CO2 等极大基数传感器的补充规则。

**维度D — 极差漂移规则：**
段内极差 (max-min) ≥ 120.0 判定为不稳定。专门抓捕"过山车式"剧烈起伏和单向严重漂移。

> 严重程度优先级：极差 > 全局STD > CV > 小均值STD
> 报告输出按优先级显示"死因参数"
```

### 2.3 硬件环境体检模块（新增 §3.6）

```
独立于气体化学通道分析，对11个环境/物理监控通道进行全实验周期的绝对健康体检：

| 通道类型 | 判定方式 | 安全范围 |
|----------|----------|----------|
| VCC供电电压 | 均值+STD | 2000~3000mV, STD<150mV |
| 温度 | 均值+STD | -10~85℃, STD<10℃ |
| 湿度 | 均值+STD | 0~100%, STD<25% |
| 气压 | 均值+STD | 700~1200mbar, STD<50mbar |

不使用CV判定（环境变量不以均值为基准），而是基于工程物理常识的绝对区间判断。
```

### 2.4 智能日期提取（更新 §2.2）

```
文件名格式兼容两种模式：
1. 标准格式：日期-ID-气体-浓度-重复次数.txt
   例：20230320-302-bingtong-30ppm-1.txt
   
2. 非标准格式（含额外前缀）：
   例：yangji1-034-huangqin(26)-2408-830-20211013_20260306094501.txt
   
系统使用正则 r'20\d{6}' 从文件名中自动提取8位连续日期数字，
确保在非标准命名下也能正确解析日期信息。
```

## 三、建议的更新方式

### 方案A：全新重写（推荐）

**做法：** 以当前代码为唯一真相源（Single Source of Truth），用 `docx-js` 重新生成一份全新的算法说明书。

**优点：**
- 与代码100%一致，无版本偏差
- 结构清晰，无历史遗留的过时内容
- 可自动化维护（代码参数变更时可同步更新文档）

**缺点：**
- 工作量大（约需1-2天）
- 原文档中的图片需要重新截取

### 方案B：增量修补

**做法：** 在原docx基础上通过XML编辑，跟踪修订（Track Changes）方式标记修改。

**优点：**
- 保留原有格式和图片
- 修改过程可追溯

**缺点：**
- 大量章节需要重写，XML编辑复杂度高
- 可能遗留未发现的旧内容

## 四、总结

当前旧说明书覆盖了约60%的代码功能。主要缺失的40%集中在：
1. 新版49传感器硬件支持
2. 四维基线判别准则
3. 硬件环境体检模块（single.py + multi.py 均已实现）
4. 自适应归一化方案
5. 多项细节修复（日期解析、描述文字、漂移警告）
6. multi.py 从 v3 到 v6 的同步升级（日期解析、零值跳过、环境体检）

**建议优先采用方案A（重写）**，以代码中的参数和逻辑为准，确保交到老师手中的文档与代码完全一致。
